A payment is usually made via these steps:

  1. The shop creates a payment resource via the API.
  2. From the payment representation that is received, the paywin link is retrieved.
  3. The customer is forwarded using the paywin link, taking them to the payment window at CertiTrade.
  4. The customer completets the payment and CertiTrade sends a callback to the shop.
  5. The customer returns to the shop via return_url.
  6. The shop retrieves the customer’s payment and verifies that it has been paid.
  7. The shop shows a receipt for the payment made.

A graphical description of the general sequence is shown in Figure 1. More detailed information on each payment method can be found in section 3.2 onward.

Callback

While the payment process is under way and the customer is in the payment window, the system reports back to the shop via the callback_url that is specified when a payment is created.

The purpose of the reporting back is to keep the shop informed of the progress of the payment. A callback is made with a POST call to callback_url. The call contains a few descriptive parameters and also a hash code that is used for verification.

The parameters are sent as POST data. The call includes the following:

Variable Description
paymentid The payment that is being referred to.
merchantid The merchant’s ID.
function The call’s function, i.e. APPROVE, DECLINE, CANCEL or ERROR.
description Code that specifies the function. See definition below.
timestamp Timestamp with the format YYYYMMDDhhmmss.
sha256code Hash code computed using hmac (sha256) and the shared secret key.

Figure 1: Sequence diagram for the general progress of the payment

The hash code is computed as follows:

apikey = [API-nyckel]
str = paymentid + merchantid + function + decscripton + timestamp
sha256code = hmac_sha256(apikey, str)

To get more detailed results, e.g. from authorization, the client can retrieve the payment resource after the payment is complete.

Type definition description

Value Function Description
2001 DECLINE Error in 3-D Secure.
2002 DECLINE Authorization failed.
2003 DECLINE Suspected fraud attempt.
2004 DECLINE The card is invalid.
3001 CANCEL The customer clicked cancel.
3002 CANCEL Timeout for the session.
4001 ERROR Max. number of payment attempts made.
4002 ERROR Fraud.
4003 ERROR The session was cancelled by the system.
4004 ERROR Error that makes it pointless to continue with the payment (e.g. in the event of an error code indicating that the merchant ID is not set up properly).

Card payments

Card payments can be made in a number of different ways. What all the payment methods involving cards have in common is what happens after authorization has been given.

When a payment is approved for capture, it will be in the state READY_FOR_CAPTURE and will be captured during the following night. Once a payment has been captured it will be set to the state CAPTURED.

A payment that is in the state WAITING_FOR_APPROVAL or READY_FOR_CAPTURE can be cancelled by setting the state to CANCELLED. The authorization on the card will then be discarded.

Card payments can be refunded.

CARD

In a CARD payment, a view is presented in which the customer can fill out his/her card details in order to make a payment. If the customer’s card and merchant are enrolled with 3-D Secure (Verified by Visa or MasterCard SecureCode) then the customer may be forwarded to his/her bank for authentication.

If the merchant has manual approval of card payments, then after successful authorization the payment resource will be set to the state WAITING_FOR_APPROVAL. The payment will then not be captured until the merchant manually approves it. This is done by setting the state of the payment resource to READY_FOR_CAPTURE.

If the merchant has automatic approval of card payments, then after successful authorization the payment resource will be set directly to the state READY_FOR_CAPTURE.

RECURRING and ONE_CLICK

In a RECURRING or ONE_CLICK payment the merchant can draw funds directly from a customer that has previously made a payment to them.

The authorization is made directly without the earlier customer being involved. If authorization is successful the payment resource is set directly to the state READY_FOR_CAPTURE. If authorization fails, the payment resource will be given the state FAILED.

In order to be able to make RECURRING or ONE_CLICK payments for a customer, a payment_account belonging to the customer must first be created and an initializing card payment made as described in 3.2.1. For RECURRING payments, a payment_account must be created before the initializing payment is made, while for ONE_CLICK payments this is done afterwards by referring to the payment made previously.

The initializing payment need not necessarily be captured but can instead then be cancelled; the important thing is simply that an authorization has been carried out.

In cases where fraud is suspected, for example, the authorization will fail and the payment_account used will be set to the state SUSPENDED. For RECURRING you can then make a new initializing payment in order to continue using the same account, while for ONE_CLICK you have to then create a new account.

Invoice payments

Invoice payments can be made via a number of participants. The overall payment process is the same for all invoice payments. When customers arrive at CertiTrade they are first presented with a view in which they can enter their civil/corporate registration number and then a view in which they confirm the purchase.

Payments by invoice can only be made by attaching a product list. These are then shown on the invoice.

If the purchase is successful the payment resource will be set to the state WAITING_FOR_APPROVAL. To create an invoice its state is then set to CAPTURED. It can also be cancelled by setting the state to CANCELLED.

INVOICE_SVEA

The payment method INVOICE_SVEA allows invoice payments via SveaWebPay.

Note that whether or not an invoice is sent out depends on the agreement with Svea. To automatically send out an invoice when a payment is set to CAPTURED there must be automatic approval of invoices at Svea. If this is not the case, it is necessary to go into Svea’s Admin and manually approve the invoice first before it is sent off.

Credit notes cannot be created for SveaWebPay invoice payments at present.

The invoice is always sent out by post.

INVOICE_PAYEX

The payment method INVOICE_PAYEX allows invoice payments via PayEx. The shop is paid immediately since PayEx buys the invoice and deals with invoicing the customer.

Invoice purchases via PayEx are only available to customers registered in Sweden and are restricted to purchases in SEK.

A credit note can be created by creating a refund resource for a CAPTURED payment.

The invoice is always sent out by post.

DIRECT_PAYEX

The payment method DIRECT_PAYEX is used for direct payments via PayEx. When the customer arrives at CertiTrade he/she is redirected to the bank that the shop has chosen and completes the payment.

If the purchase is successful, the payment state is set directly to CAPTURED. If the payment failed, it is set to FAILED. It is not possible to cancel a direct payment.

Refunds cannot be made for PayEx direct payments.

PARTPAY_PAYEX

The payment method PARTPAY_PAYEX is used for part payments via PayEx. The shop is paid immediately when the purchase is made and PayEx deals with the invoicing of a smaller partial amount monthly until the entire amount has been paid.

The payment process is the same as for invoices but instead of entering a civil/corporate registration number the address details are entered directly.

Part payments via PayEx are only available to customers registered in Sweden and are restricted to purchases in SEK.

Swish

Swish is only available in the Swedish market in SEK.

SWISH_E

The payment method SWISH_E is used for Swish payments in e-commerce. The shop needs a “Swish handel” agreement with its bank.

The payment process is similar to other methods but instead of a payment page a waiting page is displayed during checkout. To proceed from the waiting page the customer must approve the payment in the Swish app on their mobile phone.

When the payment has been completed a callback is made to the shop.

SWISH_M

The payment method SWISH_M is used for Swish payments for in-app purchases. The app provider needs a “Swish handel” agreement with its bank.

The payment process is not similar to other payment methods. When creating a SWISH_M payment a “PaymentRequestToken” is received. This token is used when opening the Swish app from the App with the in-app purchase.

When the payment has been completed a callback is made to a server specified by the app provider.
